│      │                                                      │
│      ▼                                                      │
│    ③ 用户级限流 ──→ Redis INCR (5次/秒/用户)                │
│      │                                                      │
│      ▼                                                      │
│    ④ 本地缓存 ──→ Caffeine (5-30秒)                        │
│      │                                                      │
│      ▼                                                      │
│    ⑤ Redis售罄检查 ──→ 快速拒绝                             │
│      │                                                      │
│      ▼                                                      │
│    ⑥ 重复购买检查 ──→ Redis + DB双重校验                     │
│      │                                                      │
│      ▼                                                      │
│    ⑦ Redis原子DECR ──→ 预扣库存                             │
│      │                                                      │
│      ▼                                                      │
│    ⑧ RabbitMQ异步下单 ──→ 削峰填谷                          │
│      │                                                      │
│      ▼                                                      │
│    ⑨ DB乐观锁 ──→ 最终库存扣减                              │
│      │                                                      │
│      ▼                                                      │
│    ⑩ 前端轮询结果                                            │

```
┌─────────────────────────────────────────────────────────────┐
│                   05 Redis缓存架构                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  多级缓存：Caffeine → Redis → MySQL                         │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                  Redis Key 设计                      │   │
│  ├──────────────────────┬──────────────────────────────┤   │
│  │  Key                 │  用途 / TTL                  │   │
│  ├──────────────────────┼──────────────────────────────┤   │
│  │  stock:{id}          │  秒杀库存 (永久)              │   │
│  │  goods_over:{id}     │  售罄标记 (永久)              │   │
│  │  seckill_user:{u}:{g}│  用户秒杀标记 (1小时)         │   │
│  │  goods_list:all      │  商品列表缓存 (60秒)          │   │
│  │  cart_list:{uid}     │  购物车缓存 (5分钟)           │   │
│  │  favorite_list:{uid} │  收藏列表缓存 (5分钟)         │   │
│  │  login_rate:{ip}     │  登录限流 (1分钟)             │   │
│  │  captcha:{id}        │  验证码 (5分钟)               │   │
│  └──────────────────────┴──────────────────────────────┘   │
│                                                             │
│  缓存策略：Cache-Aside (读: 缓存→DB, 写: DB→删缓存)         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 答辩注意事项

### 常见提问及参考回答

**Q1: 为什么选择Redis作为缓存而不是Memcached？**
> A: Redis支持丰富的数据结构（String、Hash、List、Set），支持持久化（RDB/AOF），支持原子操作和Lua脚本，适合秒杀场景的库存预扣减。而Memcached只支持简单的键值对，功能相对有限。

**Q2: 如何保证秒杀不超卖？**
> A: 采用多层防护：①Redis原子DECR预扣库存 ②数据库乐观锁 `WHERE stock_count > 0` ③唯一约束防止重复购买。这三层保证了即使在高并发场景下也不会超卖。

**Q3: 为什么使用消息队列异步下单？**
> A: 秒杀瞬间会有大量请求，如果同步写数据库会导致连接池耗尽。使用消息队列可以：①削峰填谷，将瞬时流量转化为队列消息 ②异步处理，提高系统吞吐量 ③重试机制，提高订单创建成功率。

**Q4: Redis和数据库的数据一致性如何保证？**
> A: 采用Cache-Aside模式：读时先查Redis再查DB，写时先更新DB再删除Redis缓存。对于秒杀库存，Redis作为预扣减层，DB作为最终一致性层，通过定时任务和订单超时机制保证最终一致。

**Q5: 分布式锁的使用场景是什么？**
> A: 在管理员补充库存、修改秒杀活动状态等操作中使用Redisson分布式锁，保证集群环境下同一时间只有一个实例能执行操作，避免并发冲突。

**Q6: 系统如何处理订单超时？**
> A: 使用`@Scheduled`定时任务每60秒扫描一次超时订单（超过15分钟未支付），自动取消订单并回滚库存（DB + Redis），同时删除秒杀成功记录允许其他用户购买。

**Q7: AI助手是如何实现的？**
> A: 集成阿里云百练平台（通义千问大模型），通过DashScope SDK调用AI接口。支持三种模型可选（极速/增强/旗舰），支持多轮对话、对话历史、消息复制和重新生成。当AI服务不可用时，降级为基于关键词匹配的规则回复。小程序端为独立全屏页面，PC端为浮动聊天窗口。
